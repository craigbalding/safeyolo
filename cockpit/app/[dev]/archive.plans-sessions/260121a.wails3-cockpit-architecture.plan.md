# Wails3 Cockpit Architecture Plan

**Date**: 2026-01-21
**Status**: Plan
**by**: Minimax3.2 in Claude CLI

## Summary

Architectural design for Wails3-based cockpit app that monitors and orchestrates Bun/Python/shell scripts on the host OS. The app provides a web-based UI (tiles) for script management and supports both UI-launched and CLI-launched scripts.

## Proposed Approach

### Communication Bridge Strategy
- **Primary**: Unix Domain Sockets (UDS) + JSON-RPC over stdio for local scripts
- **Go ↔ UI**: Wails3 native IPC events (zero serialization overhead)
- **CLI integration**: Env vars (`COCKPIT_BRIDGE`) + PID files for external processes

### Architecture Layers
1. **Catalog Layer** (`scripts.toml`): TOML-based script definitions
2. **Go Layer** (thin): Process orchestration only, no business logic
3. **Script Layer** (Bun/Python): Bridge adapters for output streaming
4. **UI Layer** (Lit WebComponents): Tile-based dashboard

### Key Design Decisions
- **Minimal dependencies**: wails, gorilla/websocket, go-toml only
- **Offline-capable**: CLI scripts work without UI via PID file discovery
- **Local-only**: UDS + stdio, no network exposure by default
- **WebComponents**: Lit for UI, generated alongside scripts

## Implementation Steps

1. Create `scripts.toml` catalog schema and example entries
2. Implement `internal/cockpit/cockpit.go` - ProcessManager, ScriptCatalog
3. Implement `internal/cockpit/wails_bridge.go` - Go→JS event emission
4. Implement `scripts/lib/bridge.ts` and `bridge.py` adapters
5. Create `frontend/src/components/cockpit-tile.ts` - Lit component
6. Create `frontend/src/components/cockpit-dashboard.ts` - Dashboard grid
7. Implement CLI wrapper for external script discovery

## Potential Risks

- **Wails3 beta**: API changes may require adjustments
- **Process isolation**: Need careful handling of zombie processes
- **Script security**: Untrusted scripts could expose host access

## Dependencies

- Wails v3 (latest beta)
- Lit (via Bun npm)
- go-toml v2
- gorilla/websocket
