# Cockpit Script Bridge Architecture

Consolidated architecture plan for the Wails3 cockpit UI with flexible script orchestration, supporting both UI-launched and CLI-started scripts with live monitoring.

## The prompt

In @cockpit/app  we are building a Wails3 based app. It's UI is a "cockpit" where in tiles the user sees separate processes running on the host (os which launched the Wails app itself). The other side of the solution are bun and python scripts which may return quick results or run endlessly, and there are also stdout/err monitoring diagnostics of a system app. From the Wails UI the user can start those scripts from a catalogue, and he monitors the output or status of those scripts in the UI. Also the user might start some scripts from CLI and expects those scripts appear on the UI as live processes and monitor them. So the integration of Wails3 app and the scripts should be flexible, and it means some sort of https (or different network) bridge with lanes between scripts and their controllers/views in the Wails managed webui. For the webapp implementation we tend to consider webcomponents -- the thinking is that when one generates a bun script it also creates the webapp unit for it, and since the business logic is not in the UI, wc/lit programing is enough, and those creating python scripts can also tell their videcoding tools to generate the wc webapp unit for that (plus the conventions of integration into the webapp). We prefer to use minimal set of 3rd party dependencies due to the vulnerability risks, instead we prefer dependencies provided by the core ecosystem. The catalog of scripts is in the filesystem, a toml or yaml from which the Wails go app can read basic properties, locations and interaction protocol. The go layer does not handle any "business" logic, it only interoperates between the scripts and their controller/view counterparts in the Webui. While the major bridge is network (rest, sync, socket, whatever -- propose!), there must be a few scripts which work even among local network connectivity issues, so there is a housekeeping/doctor script bridge which runs scripts directly and reads their stdio, but that's not the barebone part. Lets generate considerations, suggestions for this kind of implementation. Full scripts are not needed, only snippets for the cornerstone elements.

## Executive Summary

This plan addresses a **hybrid script orchestration system** where:
- Scripts (bun/python) can be started from UI **or** CLI
- All running scripts appear as live tiles in the cockpit
- Communication uses network bridge primarily (HTTP/WebSocket)
- Fallback "doctor" scripts use direct stdio for network failure scenarios
- Web components (Lit) generated alongside scripts
- Script catalog defined in TOML/YAML

## Key Requirements Mapped to Solutions

| Requirement | Solution |
|-------------|----------|
| Scripts started from UI or CLI | **Rendezvous pattern** - scripts register with Go hub on startup |
| Live process tiles | **Process registry** in Go, events to UI |
| Network bridge | **WebSocket primary** (bidirectional, real-time) |
| Fallback for network issues | **Direct stdio bridge** for housekeeping/doctor scripts |
| Web components | **Lit custom elements**, one per script type |
| Script catalog | **TOML manifest** files, Go reads and serves to UI |
| Minimal dependencies | Lit (no framework), bun native HTTP/WS, Go stdlib |

## Communication Lanes

### Lane 1: WebSocket Primary (Recommended)
**For**: Long-running scripts, real-time streaming, bidirectional control

**Protocol**: JSON messages over WebSocket
```typescript
// Message types
type ScriptMessage = 
  | { type: 'register', scriptId: string, meta: ScriptMeta }
  | { type: 'output', scriptId: string, stream: 'stdout'|'stderr', data: string }
  | { type: 'progress', scriptId: string, percent: number, stage?: string }
  | { type: 'request', scriptId: string, requestId: string, prompt: string }
  | { type: 'complete', scriptId: string, exitCode: number }
```

**Why WebSocket over SSE/HTTP**:
- Bidirectional (script can request user input)
- Single connection per script
- Bun native support (`Bun.WebSocket`, `WebSocket` client)
- Lower overhead than HTTP polling

### Lane 2: Stdio Direct (Fallback)
**For**: Housekeeping/doctor scripts that must work during network issues

**Protocol**: Newline-delimited JSON (NDJSON) over stdin/stdout
```
вҶ’ {"cmd":"run","args":["--check"]}
вҶҗ {"type":"output","line":"Checking network..."}
вҶҗ {"type":"complete","exitCode":0}
```

**Why**: 
- Works when localhost networking is broken
- Go spawns and manages directly via `os/exec`
- Simple, no ports needed

### Lane 3: HTTP REST (Optional)
**For**: One-shot queries, status checks, script-as-microservice

**Pattern**: Script runs `Bun.serve()`, Go calls it
```
GET http://localhost:{port}/status
POST http://localhost:{port}/execute
```

### 1. Go ScriptHub Service

```go
// cockpit/app/scripthub.go

package main

import (
    "encoding/json"
    "net/http"
    "sync"

    "github.com/gorilla/websocket" // minimal WS dep, or use nhooyr.io/websocket
)

type ScriptHandle struct {
    ID       string
    Meta     ScriptMeta
    Conn     *websocket.Conn // nil for stdio scripts
    Process  *os.Process     // for stdio scripts
    Status   string          // "running", "stopped", "error"
}

type ScriptMeta struct {
    Name    string `json:"name"`
    Runtime string `json:"runtime"`
    Lane    string `json:"lane"`
    Tile    string `json:"tile"`
}

type ScriptHub struct {
    mu       sync.RWMutex
    scripts  map[string]*ScriptHandle
    upgrader websocket.Upgrader
    app      *application.App // Wails app for events
}

func (h *ScriptHub) HandleWebSocket(w http.ResponseWriter, r *http.Request) {
    conn, err := h.upgrader.Upgrade(w, r, nil)
    if err != nil {
        return
    }
    defer conn.Close()

    for {
        _, msg, err := conn.ReadMessage()
        if err != nil {
            break
        }
        
        var m map[string]any
        json.Unmarshal(msg, &m)
        
        switch m["type"] {
        case "register":
            scriptID := m["scriptId"].(string)
            h.mu.Lock()
            h.scripts[scriptID] = &ScriptHandle{
                ID:     scriptID,
                Conn:   conn,
                Status: "running",
                Meta:   parseScriptMeta(m["meta"]),
            }
            h.mu.Unlock()
            
            // Notify UI
            h.app.Event.Emit("script:registered", scriptID)
            
        case "output":
            // Forward to UI
            h.app.Event.Emit("script:output", m)
            
        case "complete":
            h.mu.Lock()
            if s, ok := h.scripts[m["scriptId"].(string)]; ok {
                s.Status = "completed"
            }
            h.mu.Unlock()
            h.app.Event.Emit("script:complete", m)
        }
    }
}

// ListScripts returns all registered scripts for UI
func (h *ScriptHub) ListScripts() []ScriptHandle {
    h.mu.RLock()
    defer h.mu.RUnlock()
    
    result := make([]ScriptHandle, 0, len(h.scripts))
    for _, s := range h.scripts {
        result = append(result, *s)
    }
    return result
}
```

### 2. Bun Script Self-Registration

```typescript
// cockpit/scripts/lib/bridge.ts

const HUB_URL = process.env.COCKPIT_HUB_URL || 'ws://localhost:9100/ws';

export interface ScriptMeta {
  name: string;
  runtime: 'bun' | 'python';
  lane: 'websocket';
  tile: string;
}

export class CockpitBridge {
  private ws: WebSocket;
  private scriptId: string;

  constructor(scriptId: string, meta: ScriptMeta) {
    this.scriptId = scriptId;
    this.ws = new WebSocket(HUB_URL);
    
    this.ws.onopen = () => {
      this.send({ type: 'register', scriptId, meta });
    };
    
    this.ws.onmessage = (event) => {
      const msg = JSON.parse(event.data);
      if (msg.type === 'command') {
        this.handleCommand(msg);
      }
    };
  }

  private send(msg: object) {
    this.ws.send(JSON.stringify(msg));
  }

  output(data: string, stream: 'stdout' | 'stderr' = 'stdout') {
    this.send({ type: 'output', scriptId: this.scriptId, stream, data });
  }

  progress(percent: number, stage?: string) {
    this.send({ type: 'progress', scriptId: this.scriptId, percent, stage });
  }

  complete(exitCode: number = 0) {
    this.send({ type: 'complete', scriptId: this.scriptId, exitCode });
    this.ws.close();
  }

  // Override in script for commands from UI
  protected handleCommand(msg: { cmd: string; args?: any }) {}
}

// bridge.progress(50, 'Connecting');
// bridge.complete(0);
```

```typescript
  bridge.output(`Error: ${err.message}`, 'stderr');
  bridge.complete(1);
```

### 5. Stdio Doctor Script Pattern

```go
// cockpit/app/doctor.go - Direct stdio execution for fallback scripts

func (h *ScriptHub) RunDoctorScript(scriptPath string) error {
    cmd := exec.Command("bun", "run", scriptPath)
    
    stdin, _ := cmd.StdinPipe()
    stdout, _ := cmd.StdoutPipe()
    
    if err := cmd.Start(); err != nil {
        return err
    }
    
    // Read stdout line by line, emit events
    go func() {
        scanner := bufio.NewScanner(stdout)
        for scanner.Scan() {
            line := scanner.Text()
            var msg map[string]any
            if err := json.Unmarshal([]byte(line), &msg); err == nil {
                h.app.Event.Emit("script:output", msg)
            }
        }
    }()
    
    // Optionally send commands via stdin
    // json.NewEncoder(stdin).Encode(map[string]any{"cmd": "check"})
    
    return cmd.Wait()
}
```

### 6. Catalog Service (Go)

```go
// cockpit/app/catalogservice.go

import (
    "os"
    "github.com/BurntSushi/toml"
)

type CatalogEntry struct {
    ID          string   `toml:"id"`
    Name        string   `toml:"name"`
    Description string   `toml:"description"`
    Runtime     string   `toml:"runtime"`
    Entry       string   `toml:"entry"`
    Lane        string   `toml:"lane"`
    Tile        string   `toml:"tile"`
    Tags        []string `toml:"tags"`
    Fallback    bool     `toml:"fallback"`
}

type Catalog struct {
    Meta    struct{ Version string } `toml:"meta"`
    Scripts []CatalogEntry           `toml:"scripts"`
}

type CatalogService struct {
    catalogPath string
}

func (s *CatalogService) GetCatalog() (Catalog, error) {
    var catalog Catalog
    _, err := toml.DecodeFile(s.catalogPath, &catalog)
    return catalog, err
}

func (s *CatalogService) GetScriptsByTag(tag string) []CatalogEntry {
    catalog, _ := s.GetCatalog()
    var result []CatalogEntry
    for _, entry := range catalog.Scripts {
        for _, t := range entry.Tags {
            if t == tag {
                result = append(result, entry)
                break
            }
        }
    }
    return result
}
```

When generating a bun/python script, also generate its companion web component tile:

```
scripts/
в”ңв”Җв”Җ proxy-status.ts        # The script
в”ңв”Җв”Җ proxy-status.tile.ts   # Its web component (can be symlinked/imported into frontend)
в”ңв”Җв”Җ lib/
в”Ӯ   в””в”Җв”Җ bridge.ts          # Shared bridge library
в””в”Җв”Җ catalog.toml           # Script registry
```

**Tile naming convention**: `{script-id}.tile.ts` вҶ’ registers as `<{script-id}-tile>`

## Summary

This architecture provides:
- **Flexibility**: Scripts can self-register whether started from UI or CLI
- **Real-time**: WebSocket ensures bidirectional streaming with low latency
- **Resilience**: Stdio lane keeps housekeeping scripts working during network failures
- **Minimal deps**: Leverages Lit (existing), bun native APIs, and two small Go libraries
- **Convention over configuration**: TOML catalog and naming conventions enable script+tile co-generation
