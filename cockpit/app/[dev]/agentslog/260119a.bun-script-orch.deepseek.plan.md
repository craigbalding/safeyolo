## Approach 1: Direct Execution (os/exec)

### Core Concept
Go backend acts as parent process, spawning bun scripts via `os/exec` with real-time output streaming.

### Implementation Scenarios
1. **Interactive Script Execution**
   - UI → Go service → bun subprocess → stdout/stderr → Wails events → terminal component
   - Process management with cancellation, timeouts, resource limits

2. **Script Management & Discovery**
   - Scan predefined directories for `.ts`/`.js` files
   - Present catalog with metadata (description, parameters)
   - Editable script configurations

3. **Scheduled & Recurring Runs**
   - Cron-like scheduler in Go backend
   - Persistent job storage (SQLite/JSON)
   - UI for interval configuration

4. **Bidirectional Interactive Scripts**
   - Scripts prompting for user input
   - Events to UI → response passed back via stdin
   - Two-way communication channel

5. **Result Persistence & Audit**
   - Store script outputs, execution times, exit codes
   - Historical review with filtering/search
   - Export capabilities

6. **Security & Sandboxing**
   - Script validation before execution
   - Filesystem/network access restrictions
   - Timeout enforcement
   - Non-privileged user execution

7. **SafeYolo Integration**
   - Scripts to control SafeYolo proxy (start/stop, config updates)
   - Metrics collection via admin API
   - Log tailing and analysis

8. **Plugin System**
   - Extensible script directory with metadata files
   - UI auto-discovers new scripts
   - Custom UI extensions for script types

### Technical Considerations
- **Path resolution**: Ensure bun is in `PATH` or configure absolute path
- **Concurrent execution**: Manage multiple script processes with unique IDs
- **Error recovery**: Handle script crashes, partial output, cleanup
- **UI components**: Terminal emulator (xterm.js), progress indicators, log viewer

## Approach 2: HTTP-Based Async RPC Integration

### Core Concept
Leverage bun's HTTP capabilities and Go's HTTP interface for decoupled, peer-to-peer communication.

### Implementation Scenarios

#### 1. Bun‑as‑Server Pattern
- **Script‑hosted HTTP endpoints**: Each bun script exposes lightweight HTTP server (`bun serve`) on dynamically assigned ports
- **Go backend as orchestrator**: Discovers/registers script endpoints via service registry (file‑based or in‑memory)
- **Frontend indirect control**: UI → Go service → HTTP request → bun script → results stream back via Wails events

#### 2. Bun‑as‑Client Pattern
- **Scripts as HTTP clients**: Bun scripts periodically poll Go's HTTP API for queued tasks
- **Go‑managed work queue**: Maintain task queue, assign work to registered bun workers
- **Push notifications**: Use Server‑Sent Events (SSE) to notify bun clients of pending tasks

#### 3. Bidirectional WebSocket Bridge
- **Persistent connection**: Bun script establishes WebSocket to Go backend
- **RPC over messages**: JSON‑RPC or custom protocol for method invocation
- **Real‑time duplex**: Both sides can invoke methods and stream results

#### 4. HTTP‑Hook Event System
- **Webhook‑based triggering**: Bun scripts expose callback URLs that Go calls on UI events
- **Event‑driven choreography**: Scripts subscribe to specific event types (e.g., `proxy.start`, `log.tail`)
- **Loose coupling**: Scripts can be restarted independently; Go retries failed webhooks

#### 5. Shared‑Port Co‑hosting
- **Unified HTTP server**: Go exposes `/api/scripts/*` endpoints that proxy to bun subprocesses
- **Port sharing**: Single Wails HTTP server handles both UI assets and script API routes
- **Built‑in auth**: Leverage Wails' existing security context for all requests

### Architectural Advantages
- **Process isolation**: Bun crashes don't bring down Go backend
- **Polyglot future‑proofing**: Same HTTP interface could support Python/Node scripts
- **Independent development**: Scripts can be tested via curl/Postman without UI
- **Scalability**: Multiple script instances can load‑balance via Go router
- **Restart resilience**: Scripts can be updated/restarted without disrupting the orchestrator

### Implementation Considerations
- **Service discovery**: Need registry for script endpoints (port files, HTTP registration endpoints)
- **Authentication**: Shared secret or JWT between Go and bun processes
- **Error handling**: HTTP retries, circuit breakers, timeouts, health checks
- **Resource management**: Go must track and cleanup orphaned bun HTTP servers
- **Port management**: Dynamic port allocation, conflict resolution
- **Security**: Input validation, rate limiting, CORS configuration

## Comparison & Recommendations

| Aspect | Direct Execution (os/exec) | HTTP‑Based Integration |
|--------|----------------------------|------------------------|
| **Complexity** | Lower (standard parent‑child) | Higher (distributed coordination) |
| **Isolation** | Process boundaries, but shared lifecycle | Complete process independence |
| **Scalability** | Limited by Go process management | Highly scalable (multiple script instances) |
| **Real‑time streaming** | Native via pipes | Requires WebSocket/SSE layer |
| **Error resilience** | Script crash affects Go context | Script crash isolated, Go can restart |
| **Development/testing** | Requires full Wails context | Scripts testable standalone via HTTP |
| **Security** | Easier sandboxing (chroot, cgroups) | Network‑based security (firewall, auth) |
| **UI integration** | Direct event streaming | Indirect via Go proxy/events |

### Recommended Hybrid Approach
1. **Start with Direct Execution** for simplicity and rapid prototyping
2. **Evolve to HTTP‑based** for long‑running, complex, or multiple script scenarios
3. **Design service interfaces** that can work with both patterns (unified API)

## Notes

- The Wails3 architecture already provides strong foundation with service binding and event system
- bun 1.3+ HTTP capabilities (server, client, WebSocket) align well with Go's net/http package
- SafeYolo integration should consider proxy's own admin API (HTTP) for consistent patterns
- Both approaches can coexist: simple scripts via direct execution, complex services via HTTP
