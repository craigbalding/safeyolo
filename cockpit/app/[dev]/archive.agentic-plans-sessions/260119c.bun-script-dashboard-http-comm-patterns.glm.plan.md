## Scenarios Analyzed

### Part 1: General Implementation Patterns

#### 1. Simple Direct Execution (synchronous)
- **Pattern**: Frontend → Go method → os/exec(bun) → stdout capture → return
- **Use case**: Quick operations (< 5s)
- **Pros**: Simplest
- **Cons**: UI freezes for long scripts

#### 2. Asynchronous Job Queue with polling
- **Pattern**: Go spawns goroutine, returns Job ID, frontend polls status
- **Use case**: Non-blocking, concurrent scripts
- **Pros**: Non-blocking, cancelable
- **Cons**: Polling overhead, stale data possible

#### 3. Real-time Streaming via Wails events
- **Pattern**: Go spawns bun with pipes → goroutine reads lines → EmitEvent() → frontend listens
- **Use case**: Real-time terminal-like output
- **Pros**: Real-time, no polling
- **Reference**: main.go:24 (event registration), my-element.ts:29 (event listener)

#### 4. Hybrid: Streaming + Job Management
- **Pattern**: Stream via events + maintain job state map
- **Use case**: Production dashboard
- **Pros**: Best UX - real-time + history
- **Pros**: Cancelable jobs

#### 5. Bidirectional Communication (interactive)
- **Pattern**: Bun makes HTTP request to localhost → Go prompts user → Go sends response
- **Use case**: Interactive CLI-like scripts
- **Pros**: Mid-execution user input
- **Cons**: Requires Go HTTP server + bun script modifications

#### 6. Script Registry & Discovery
- **Pattern**: bun_scripts/ folder → Go scans → returns available scripts
- **Use case**: Dynamic script discovery
- **Pros**: Script versioning/metadata
- **Leverages**: PatternFly components (pf-card, pf-button)

#### 7. Parameterized Execution
- **Pattern**: Frontend form → Go constructs bun command with flags → executes
- **Use case**: Type-safe parameter handling
- **Pros**: Input validation before execution

#### 8. Execution History & Re-run
- **Pattern**: Go stores execution log in SQLite → frontend shows history
- **Use case**: Audit trail, quick re-run
- **Pros**: Debugging past runs

### Part 2: HTTP-Based Communication Patterns

#### 1. Go HTTP Server → Bun Client (Script Control)
- **Pattern**: Bun fetches Go's status/control endpoint
- **Use cases**: Pause/resume/kill, parameter injection, progress queries
- **Pros**: Simple fetch, Go maintains control
- **Cons**: Polling-based, requires script modifications

#### 2. Bun HTTP Server → Go Client (Script Services)
- **Pattern**: Bun spawns server, Go acts as HTTP client
- **Use cases**: Metrics endpoints, command acceptance, long-running workers
- **Pros**: Bun can push updates, script controls lifecycle
- **Cons**: Port management conflicts

#### 3. Bidirectional HTTP with Heartbeat (Durable Execution)
- **Pattern**: POST /job/start, periodic POST /job/heartbeat, POST /job/complete
- **Use cases**: Long-running scripts (hours/days), fault-tolerant jobs
- **Pros**: Survives network blips, job state in Go
- **Cons**: More complex, heartbeat tuning

#### 4. WebSocket Communication (Real-time Bi-directional)
- **Pattern**: bun → ws://localhost:9000/ws, both send messages anytime
- **Use cases**: Terminal streaming, interactive scripts, live collaboration
- **Pros**: True real-time, low latency, full duplex
- **Cons**: More complex, need reconnection logic
- **Note**: Bun v1.3+ supports Bun.WebSocket

#### 5. Server-Sent Events (SSE) (One-way Streaming)
- **Pattern**: bun GET /job/events → Go streams event-stream
- **Use cases**: Bun pulls logs/updates, Go orchestrates multiple scripts
- **Pros**: Simpler than WebSocket, standard HTTP, built-in reconnection
- **Cons**: One-way only (Go → Bun)

#### 6. HTTP Long Polling (Near Real-time)
- **Pattern**: bun GET /job/wait (hangs) → Go holds until event or timeout
- **Use cases**: WebSocket alternative, network restrictions
- **Pros**: Works everywhere, near real-time
- **Cons**: Connection overhead, not truly real-time

#### 7. Bun as Microservice (REST API Pattern)
- **Pattern**: Bun.serve() with REST routes → Go acts as API gateway
- **Use cases**: Long-running daemons, service mesh UI
- **Pros**: Scripts are standalone services, testable independently
- **Cons**: Port discovery needed, lifecycle complexity

#### 8. Webhook Pattern (Event-Driven)
- **Pattern**: Bun POSTs to configured webhook URL on events
- **Use cases**: Third-party script integration, decoupled architecture
- **Pros**: Scripts don't depend on Go, multiple endpoints
- **Cons**: No delivery guarantee, can't query script state

#### 9. HTTP Chunked Transfer (Large Output Streaming)
- **Pattern**: Chunked encoding for large files/logs
- **Use cases**: Large artifact generation, backup/restore
- **Pros**: Memory efficient, real-time streaming
- **Cons**: Frontend progressive rendering needed

#### 10. Multipart Scenarios (Mixed Communication)
- **Pattern**: Combines multiple patterns (SSE + WebSocket + polling)
- **Use cases**: Production systems, complex scripts
- **Pros**: Best of all worlds, phase-specific optimization
- **Cons**: Most complex, multiple connection management

## Key Components Identified

### Go Backend
- `ScriptService` (like GreetService) with methods:
  - `Execute()`, `ListScripts()`, `CancelJob()`, `GetHistory()`
- Job manager: map[jobID]*Job struct with state, pid, output buffer
- `os/exec` with `Cmd.StdoutPipe()` for streaming

### Frontend (Lit + PatternFly)
- Job list component (pf-card, pf-button)
- Terminal output component (scrolling div with streaming text)
- Script discovery UI (grid of available scripts)
- Parameter forms for each script

### Bun Scripts
- Standard CLI scripts (unmodified for simple scenarios)
- Optional: HTTP client for interactive scenarios
- Optional: Bun.serve() for service pattern
