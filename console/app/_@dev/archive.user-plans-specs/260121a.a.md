In @cockpit/app  we are building a Wails3 based app. 

It's UI is a "cockpit" where in tiles the user sees separate processes running on the host (host = the os which launched the Wails app itself). The actual design of the Webapp is not important at this iteration.

The other side of the solution are:
    - bun and python scripts which may return quick results or run endlessly, 
    - one or two binary apps which spit to stdout/err monitoring diagnostics. 

*There's a catzZlsllsalog*

From the Wails UI the user can start those scripts from a catalogue, and he monitors the output or status of those scripts in the UI. Also the user might start some scripts from CLI and expects those scripts appear on the UI as live processes and monitor them. So the integration of Wails3 app and the scripts should be flexible, and it means some sort of https (or different network) bridge with lanes between scripts and their controllers/views in the Wails managed webui. For the webapp implementation we tend to consider webcomponents -- the thinking is that when one generates a bun script it also creates the webapp unit for it, and since the business logic is not in the UI, wc/lit programing is enough, and those creating python scripts can also tell their videcoding tools to generate the wc webapp unit for that (plus the conventions of integration into the webapp). We prefer to use minimal set of 3rd party dependencies due to the vulnerability risks, instead we prefer dependencies provided by the core ecosystem. The catalog of scripts is in the filesystem, a toml or yaml from which the Wails go app can read basic properties, locations and interaction protocol. The go layer does not handle any "business" logic, it only interoperates between the scripts and their controller/view counterparts in the Webui. While the major bridge is network (rest, sync, socket, whatever -- propose!), there must be a few scripts which work even among local network connectivity issues, so there is a housekeeping/doctor script bridge which runs scripts directly and reads their stdio, but that's not the barebone part. Lets generate considerations, suggestions for this kind of implementation. Full scripts are not needed, only snippets for the cornerstone elements.
