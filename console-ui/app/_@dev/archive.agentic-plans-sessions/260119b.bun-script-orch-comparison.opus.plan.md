## Executive Summary

Six distinct integration patterns exist for Go ↔ Bun communication in the cockpit dashboard. Each trades off complexity, performance, isolation, and real-time capabilities differently.

| Approach | Complexity | Real-time | Isolation | Best For |
|----------|------------|-----------|-----------|----------|
| 1. Direct Execution (os/exec) | Low | ★★★ | Medium | Quick scripts, prototyping |
| 2. HTTP-Based RPC | Medium | ★★ | High | Long-running services |
| 3. Stdio JSON-RPC | Low-Med | ★★★ | Medium | Structured bidirectional |
| 4. Unix Domain Sockets | Medium | ★★★ | High | High-throughput local |
| 5. File-Based Queue | Low | ★ | High | Restricted environments |
| 6. Embedded JS Runtime | High | ★★★ | Low | No external deps |

## Approach 1: Direct Execution (os/exec)

### Pattern
```
UI → Wails Event → Go Service → os/exec.Command("bun", "run", script.ts)
                                      ↓
                              stdout/stderr pipes
                                      ↓
                              Wails Event → UI
```

### Characteristics
- **Startup**: New process per execution
- **Communication**: Pipes (stdout, stderr, stdin)
- **Lifecycle**: Tied to script execution duration
- **Streaming**: Line-buffered output via goroutine readers

### Pros
- Simplest implementation
- Native process control (signals, kill, timeout)
- No additional protocol overhead
- Familiar subprocess pattern

### Cons
- Process spawn overhead per invocation
- Unstructured output (text parsing required)
- No built-in request/response correlation
- Limited bidirectional interaction

### When to Use
- One-shot scripts with simple output
- Quick prototyping phase
- Scripts that run and exit (no long-lived state)

## Approach 2: HTTP-Based Async RPC

### Pattern
```
UI → Go Service → HTTP Request → Bun HTTP Server (localhost:PORT)
                       ↓
              JSON Response / SSE Stream
                       ↓
              Wails Event → UI
```

### Variants
- **Bun-as-Server**: Script exposes HTTP endpoints, Go calls them
- **Bun-as-Client**: Script polls Go's task queue
- **WebSocket Bridge**: Persistent bidirectional connection

### Characteristics
- **Startup**: Script runs as long-lived HTTP server
- **Communication**: HTTP/WebSocket (JSON payloads)
- **Lifecycle**: Independent of individual requests
- **Streaming**: SSE or WebSocket for real-time

### Pros
- Process isolation (crashes don't affect Go)
- Standard protocols (curl-testable)
- Polyglot-ready (same interface for Python, Node)
- Scales to multiple instances

### Cons
- Port management complexity
- Service discovery overhead
- HTTP parsing overhead
- More moving parts

### When to Use
- Long-running script services
- Scripts that maintain state between calls
- When crash isolation is critical
- Multi-script orchestration scenarios

## Approach 3: Stdio JSON-RPC (LSP-Style)

### Pattern
```
UI → Go Service → subprocess stdin → Bun Script
                            ↓
                    JSON-RPC 2.0 messages
                            ↓
              subprocess stdout → Go Service → UI
```

### Protocol Example
```json
→ {"jsonrpc":"2.0","method":"runTask","params":{"task":"build"},"id":1}
← {"jsonrpc":"2.0","result":{"status":"complete","time":1234},"id":1}
← {"jsonrpc":"2.0","method":"progress","params":{"percent":50}}
```

### Characteristics
- **Startup**: Long-lived subprocess
- **Communication**: Structured JSON over stdio
- **Lifecycle**: Persistent, reusable connection
- **Streaming**: Notification messages (no id)

### Pros
- Proven pattern (LSP, DAP, MCP all use this)
- No network stack, lower latency than HTTP
- Strongly typed request/response
- Bidirectional notifications
- Single process, no port conflicts

### Cons
- Requires protocol implementation both sides
- Message framing needed (Content-Length headers or newline-delimited)
- Debugging harder than HTTP (no curl)
- Process crash kills connection

### When to Use
- Structured bidirectional communication needed
- Lower latency than HTTP required
- Want to avoid network/port complexity
- Modeling after LSP/MCP patterns

## Approach 4: Unix Domain Sockets

### Pattern
```
UI → Go Service → UDS Connect → /tmp/script-{id}.sock ← Bun Server
                        ↓
                 HTTP-over-UDS or raw frames
                        ↓
              Response → Go Service → UI
```

### Characteristics
- **Startup**: Script creates socket file
- **Communication**: Like TCP but file-based
- **Lifecycle**: Socket exists while script runs
- **Streaming**: Full TCP-like capabilities

### Pros
- Faster than localhost TCP (kernel bypass)
- File-based security (permissions)
- No port conflicts possible
- Can use HTTP client/server over UDS

### Cons
- Unix-only (no Windows support)
- Socket file cleanup on crash
- Service discovery via filesystem
- Slightly more complex than pure TCP

### When to Use
- High-throughput local communication
- Want HTTP semantics without ports
- Security via filesystem permissions
- Linux/macOS only deployment

---

## Approach 5: File-Based Queue

### Pattern
```
UI → Go Service → write /tasks/task-{id}.json
                              ↓
                    Bun script watches directory
                              ↓
                    write /results/task-{id}.json
                              ↓
              Go reads result → Wails Event → UI
```

### Characteristics
- **Startup**: Script runs as file watcher
- **Communication**: JSON files in watched directories
- **Lifecycle**: Independent polling loop
- **Streaming**: Multiple partial result files

### Pros
- Zero network dependencies
- Works in restricted environments
- Simple debugging (inspect files)
- Natural persistence/recovery
- Maximum isolation

### Cons
- High latency (polling interval)
- File I/O overhead
- Cleanup complexity
- No true streaming

### When to Use
- Network-restricted environments
- Task queuing with persistence
- Debugging/audit trail needed
- Very loose coupling required

---

## Approach 6: Embedded JS Runtime

### Pattern
```
UI → Go Service → goja.RunString(script) → Result → UI
                     (in-process)
```

### Options
- **goja**: Pure Go ES5.1+ interpreter
- **v8go**: V8 bindings for Go
- **quickjs-go**: QuickJS bindings

### Characteristics
- **Startup**: None (in-process)
- **Communication**: Direct function calls
- **Lifecycle**: Tied to Go process
- **Streaming**: Callback functions

### Pros
- Zero subprocess overhead
- Synchronous integration
- No IPC complexity
- Single binary deployment

### Cons
- Loses bun-specific APIs (fetch, Bun.serve, etc.)
- Limited ES module support
- Security: scripts share Go memory
- Performance varies by runtime

### When to Use
- Simple scripts without bun-specific features
- Want single-binary deployment
- Sandboxed expression evaluation
- Performance-critical inline transforms

## Recommended Implementation Path

### Phase 1: Foundation (os/exec)
Start with Approach 1 for rapid iteration:
- Implement `ScriptService` with `Run`, `Stop`, `GetOutput`
- Stream stdout/stderr via Wails events
- Basic process management

### Phase 2: Structure (Stdio JSON-RPC)
Evolve to Approach 3 for complex scripts:
- Define JSON-RPC message types
- Implement message framing in Go and TypeScript helpers for bun
- Support notifications for progress/logs

### Phase 3: Scale (HTTP when needed)
Add Approach 2 for specific use cases:
- Long-running script services
- Scripts that need independent lifecycle
- Multi-instance scenarios

### Design Principle
Abstract behind `ScriptRunner` interface:
```go
type ScriptRunner interface {
    Start(script string, args []string) (ScriptHandle, error)
    Send(handle ScriptHandle, message []byte) error
    Receive(handle ScriptHandle) (<-chan []byte, error)
    Stop(handle ScriptHandle) error
}
```

This allows swapping implementations without changing callers.
