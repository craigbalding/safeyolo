# Cockpit App - Script Monitoring Architecture

**Date**: 2026-01-20
**Status**: 📋 Architecture Considerations

## The prompt

In @cockpit/app  we are building a Wails3 based app. It's UI is a "cockpit" where in tiles the user sees separate processes running on the host (os which launched the Wails app itself). The other side of the solution are bun and python scripts which may return quick results or run endlessly, and there are also stdout/err monitoring diagnostics of a system app. From the Wails UI the user can start those scripts from a catalogue, and he monitors the output or status of those scripts in the UI. Also the user might start some scripts from CLI and expects those scripts appear on the UI as live processes and monitor them. So the integration of Wails3 app and the scripts should be flexible, and it means some sort of https (or different network) bridge with lanes between scripts and their controllers/views in the Wails managed webui. For the webapp implementation we tend to consider webcomponents -- the thinking is that when one generates a bun script it also creates the webapp unit for it, and since the business logic is not in the UI, wc/lit programing is enough, and those creating python scripts can also tell their videcoding tools to generate the wc webapp unit for that (plus the conventions of integration into the webapp). We prefer to use minimal set of 3rd party dependencies due to the vulnerability risks, instead we prefer dependencies provided by the core ecosystem. The catalog of scripts is in the filesystem, a toml or yaml from which the Wails go app can read basic properties, locations and interaction protocol. The go layer does not handle any "business" logic, it only interoperates between the scripts and their controller/view counterparts in the Webui. While the major bridge is network (rest, sync, socket, whatever -- propose!), there must be a few scripts which work even among local network connectivity issues, so there is a housekeeping/doctor script bridge which runs scripts directly and reads their stdio, but that's not the barebone part. Lets generate considerations, suggestions for this kind of implementation. Full scripts are not needed, only snippets for the cornerstone elements.


## Summary

Architecture proposal for a Wails3-based "cockpit" UI that monitors and controls Bun/Python scripts and system processes. The system uses network bridges for communication between scripts and UI, with web components (Lit) as the UI building blocks.

## Core Architecture Decisions

### 1. Communication Protocol Comparison

#### Option A: gRPC (Recommended for Primary Bridge)
**Pros:**
- Type-safe, bidirectional streaming for real-time output
- Built-in code generation for Go, Bun (connect-bun), Python (grpcio)
- Efficient binary protocol, smaller overhead than JSON
- Server reflection for discovery
- Native HTTP/2 support

**Cons:**
- Requires proto file maintenance
- Slightly more complex setup
- Requires external dependencies (minimal: grpc/grpc-go, connect, etc.)

**Use case:** Real-time streaming of script stdout/stderr, bidirectional control signals.

#### Option B: WebSocket (Fallback Option)
**Pros:**
- Native browser support (no additional deps for UI)
- Simple message format (JSON text)
- Easy to debug
- Wails3 can proxy connections

**Cons:**
- No type safety (requires manual schema)
- Reconnection logic needed
- Less efficient than binary

**Use case:** Simple one-way monitoring, scripts that don't need complex control.

#### Option C: REST Polling (Simplest)
**Pros:**
- Zero additional dependencies
- Simplest to implement
- Works through any HTTP client

**Cons:**
- Not real-time (polling delay)
- High overhead for streaming
- Not suitable for interactive control

**Use case:** Scripts with infrequent status updates, health checks only.

### 2. Recommended Hybrid Approach

```
┌─────────────────────────────────────────────────────────────────┐
│                          Wails3 Go Layer                          │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐  │
│  │  gRPC Server    │  │  WebSocket Proxy│  │  Catalog Reader │  │
│  │  (Port 50051)   │  │  (Port 50052)   │  │  (Filesystem)   │  │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘  │
└─────────────────────────────┬─────────────────────────────────────┘
                              │
                              │ Network Bridge (HTTP/2, WS, or Direct Exec)
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
┌───────▼────────┐   ┌──────▼──────┐   ┌─────────▼────────┐
│ Bun Script 1   │   │ Bun Script 2│   │ Python Script    │
│ (gRPC server)  │   │ (WS server) │   │ (HTTP REST)      │
└────────────────┘   └─────────────┘   └──────────────────┘
```

**Primary Bridge:** gRPC for most scripts (streaming, control, type safety)
**Secondary Bridge:** WebSocket for scripts that need browser-native support
**Fallback:** Direct execution for housekeeping/doctor script (no network)

## 3. Data Flow Design

### Script Registration Flow
```
1. Wails3 startup
   └─> Read catalog.yaml from filesystem
   └─> For each script:
       ├─> If script not running: show "available" tile
       └─> If script running (detected via pid file or health check): show "running" tile
           └─> Subscribe to script's status endpoint
```

### Script Discovery (CLI-launched scripts)
```
1. Bun/Python script starts (either from CLI or Wails)
   └─> Script writes PID to /tmp/cockpit/${script_name}.pid
   └─> Script advertises its endpoint:
       ├─> gRPC: write to /tmp/cockpit/${script_name}.endpoint.grpc
       ├─> WS: write to /tmp/cockpit/${script_name}.endpoint.ws
       └─> REST: write to /tmp/cockpit/${script_name}.endpoint.rest
2. Wails3 Go layer periodically scans /tmp/cockpit/
   └─> Detects new PID files
   └─> Reads endpoint files
   └─> Adds to UI as "discovered" process tile
```

### Output Streaming Flow
```
Script Process
   │
   ├─> Stdout/Stderr ──> Script's gRPC server (stream) ──> Go gRPC Client ──> Wails Event ──> WC Component
   └─> Status events ──> gRPC/WS/REST ──> Go Layer ──> Wails Event ──> WC Component
```

// discoveryLoop periodically scans for new/changed scripts
func (bm *BridgeManager) discoveryLoop(ctx context.Context) {
	ticker := time.NewTicker(2 * time.Second)
	defer ticker.Stop()

	for {
		select {
		case <-ctx.Done():
			return
		case <-ticker.C:
			bm.scanForScripts()
		}
	}

// scanForScripts checks PID files and endpoint files for script discovery
func (bm *BridgeManager) scanForScripts() {
	entries, err := os.ReadDir(bm.pidDir)
	if err != nil {
		return
	}

	for _, entry := range entries {
		if entry.IsDir() {
			continue
		}

		name := entry.Name()
		// Check for PID files: ${script_name}.pid
		if filepath.Ext(name) == ".pid" {
			scriptID := name[:len(name)-4]
			bm.processDiscoveredScript(scriptID)
		}
	}
}

// processDiscoveredScript handles a discovered script
func (bm *BridgeManager) processDiscoveredScript(scriptID string) {
	bm.mu.Lock()
	defer bm.mu.Unlock()

	// Check if we already know about this script
	if info, exists := bm.scripts[scriptID]; exists {
		// Update last update time
		info.LastUpdate = time.Now()
		return
	}

	// Read PID file
	pidFile := filepath.Join(bm.pidDir, scriptID+".pid")
	pidBytes, err := os.ReadFile(pidFile)
	if err != nil {
		return
	}

	var pid int
	if _, err := fmt.Sscanf(string(pidBytes), "%d", &pid); err != nil {
		return
	}

	// Try to discover endpoint
	// Check gRPC endpoint file
	grpcEndpointFile := filepath.Join(bm.pidDir, scriptID+".endpoint.grpc")
	bridgeType := "direct"
	endpoint := ""

	if data, err := os.ReadFile(grpcEndpointFile); err == nil {
		bridgeType = "grpc"
		endpoint = string(data)
		// Connect to gRPC server
		if conn, err := grpc.NewClient(endpoint, grpc.WithTransportCredentials(insecure.NewCredentials())); err == nil {
			bm.grpcClients[scriptID] = conn
		}
	}

	// Create script info
	info := &ScriptInfo{
		ID:         scriptID,
		State:      StateRunning,
		PID:        pid,
		Endpoint:   endpoint,
		BridgeType: bridgeType,
		StartTime:  time.Now(),
		LastUpdate: time.Now(),
		Metrics:    make(map[string]MetricValue),
	}

	bm.scripts[scriptID] = info

	// Emit script discovered event
	bm.wailsEmitter.Emit("script:discovered", info)

	// Start subscribing to script events
	bm.subscribeToScript(scriptID, bridgeType, endpoint)
}

	}
}

# scripts/simple_script.py
import os
import sys
import time
import signal
import atexit

class ScriptBridge:
    """Minimal bridge for Python scripts without network dependency."""

    def __init__(self, script_name: str):
        self.script_name = script_name
        self.pid_file = f"/tmp/cockpit/{script_name}.pid"
        self.out_file = f"/tmp/cockpit/{script_name}.out"

        # Register cleanup on exit
        atexit.register(self.cleanup)
        signal.signal(signal.SIGTERM, self._signal_handler)
        signal.signal(signal.SIGINT, self._signal_handler)

        # Write PID for discovery
        with open(self.pid_file, 'w') as f:
            f.write(str(os.getpid()))

        # Clear output file
        if os.path.exists(self.out_file):
            os.remove(self.out_file)

    def _signal_handler(self, signum, frame):
        self.cleanup()
        sys.exit(0)

    def cleanup(self):
        """Remove PID file on exit."""
        try:
            if os.path.exists(self.pid_file):
                os.remove(self.pid_file)
        except Exception:
            pass

    def log(self, level: str, message: str):
        """Write log to output file for monitoring."""
        timestamp = time.strftime("%Y-%m-%dT%H:%M:%S")
        log_line = f"{timestamp} [{level}] {message}\n"

        with open(self.out_file, 'a') as f:
            f.write(log_line)

    def log_info(self, message: str):
        self.log('info', message)

    def log_warn(self, message: str):
        self.log('warn', message)

    def log_error(self, message: str):
        self.log('error', message)

  timestamp: string

## 10. Key Considerations & Recommendations

### Security
- **PID files**: Use permissions (0600) to prevent tampering
- **Network ports**: Bind to `127.0.0.1` only (localhost)
- **gRPC**: Use TLS in production (can be self-signed)
- **Authorization**: Consider API keys for scripts exposing control endpoints

### Reliability
- **Reconnection**: Auto-reconnect gRPC/WS connections on disconnect
- **Graceful shutdown**: Scripts must handle SIGTERM, cleanup PID files
- **Health checks**: Periodic health pings, mark scripts as stopped if unreachable
- **Buffering**: Go layer should buffer events, not lose logs

### Performance
- **gRPC streaming**: More efficient than polling for high-volume logs
- **Log rotation**: Implement max log lines per script (e.g., 1000 lines in memory)
- **Batch metrics**: Send metrics in batches every 1-2 seconds

### Dependencies
**Minimal external deps:**
- Go: `google.golang.org/grpc`, `connectrpc/connect-go`
- Bun: `@connectrpc/connect`, `@connectrpc/connect-fastify`
- TypeScript: `lit` (already Wails3 standard)
- Python: Standard library only for direct bridge

**Avoid:**
- Complex ORMs, query builders
- Authentication libraries (use simple token-based if needed)
- Heavy monitoring/logging frameworks
