Its Wails3 app development with bun and typescript. Now I try to choose protocols and basic patterns -- poc stage. The feature in question is the following, there is a Wails3 (alpha) app which among other things displays the output of certain terminal utilities (Linux and Maos is our concern for the poc). Those utilities are not aware of our monitoring solution. Like lets take 'ls' and 'top' linux commands as subjects for the current poc definition (one being a one-off and normal stdout thing, the other being resident and using some more complicated tui mechanics). The use case is that the user start those utilities indirectly, as being wrapped into the runner element of our solution. The runner element is a bun script which strictly utilizes only built-in bun capabilities provided by the v1.3+ level Bun API. The script must not implement any logic which is possible to delegate to the "server" side of the runner, which latter is provided by the Wails3 app, it's frontend TS layer listening on a port or the Go layer listening with any other appropriate protocol. In the use case then the command is like:

./path/runner.ts --some-option --- ls -la ./

The script has a shim saying that it has to be run with bun. The part after '---' is like the usual pattern of running something with a wrapper shell like "sh -- ls -la ./"

The only expectation regarding the setup is that the user has bun 1.3 in the PATH, and our Wails3 app is listening. Also the runner element is to be a dead simple code, which our users can read and see it has no tricky code in it, it's safe to run. What protocol should our runner element and server talk, considering that like in case with 'top' it needs to a be persistent reading of the utility's output to mirror it in our webapp on the  Wails3 side. (The Wails3 UI part is out of scope now.)
